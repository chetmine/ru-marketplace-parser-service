# Code Review Report

## 1. Архитектура и Dependency Injection

### Проблема: Изолированный DI контейнер в `ParserRegistry`
Класс `ParserRegistry` создает собственный экземпляр контейнера `awilix`, независимый от основного контейнера приложения (`src/utils/container.ts`).
- **Риск:** Парсеры, регистрируемые внутри `ParserRegistry`, изолированы от основных зависимостей приложения (Logger, Config, ProxyService и т.д.). Если парсеру потребуется доступ к базе данных или конфигурации, передать их будет сложно. Это нарушает принцип Inversion of Control в рамках всего приложения.
- **Решение:** Регистрировать классы парсеров в основном контейнере приложения и передавать их в `ParserRegistry` через конструктор (например, как массив или объект-карту). Либо передавать основной контейнер в `ParserRegistry`.

### Проблема: Локальное состояние в `BrowserService` (Stateful Service)
`BrowserService` хранит активные контексты браузера в локальной переменной `this.contexts = new Map()`.
- **Риск:** Это делает сервис "Stateful". При горизонтальном масштабировании (запуск нескольких реплик сервиса) запросы от одного `session-id` могут попадать на разные инстансы. Это приведет к тому, что каждый инстанс будет создавать свой контекст браузера, перезаписывая состояние в Redis. Это вызовет рассинхронизацию сессий (куки, localStorage) и неэффективное использование ресурсов.
- **Решение:** Если планируется масштабирование, необходимо использовать Sticky Sessions на балансировщике нагрузки или вынести управление браузерами в отдельный сервис-оркестратор.

## 2. Логика работы и Управление ресурсами

### Проблема: Неэффективная ротация прокси (`ProxyHandler`)
В `ProxyHandler`, при получении события `proxy-replaced`, происходит только обновление данных в Redis через `browserService.save()`.
- **Риск:** Активный `BrowserContext`, который уже загружен в память `BrowserService`, **не обновляется**. Playwright не поддерживает смену прокси "на лету" для уже созданного контекста. В итоге, если прокси умер во время сессии, сервис продолжит пытаться использовать мертвый прокси до тех пор, пока контекст не будет выгружен из памяти (по таймауту TTL).
- **Решение:** В `ProxyHandler` необходимо принудительно закрывать текущий контекст: `await this.browserService.closeContext(sessionId)`. Это заставит сервис создать новый контекст с новым рабочим прокси при следующем запросе пользователя.

### Проблема: Утечка ресурсов в `ProductAggregatorService`
В методе `searchProducts` закрытие страниц происходит синхронно через `forEach`, без ожидания завершения операции:
```typescript
pages.forEach(page => { page.close() })
```
- **Риск:** Метод `page.close()` — асинхронный. Функция `searchProducts` может вернуть результат и завершиться до того, как страницы реально закроются. При высокой нагрузке или ошибках это может привести к накоплению "зомби" страниц в браузере, утечке памяти и падению производительности.
- **Решение:** Использовать `await Promise.all(pages.map(page => page.close()))`.

### Проблема: Критическая ошибка (Crash) при пустом поиске
В методе `searchProductDetailed` используется `reduce` без начального значения:
```typescript
const objectWithMostFeatures = detailedProducts.reduce((max, current) => ...)
```
- **Риск:** Если по запросу не найдено ни одного продукта (массив `detailedProducts` пуст), этот код выбросит исключение `TypeError: Reduce of empty array with no initial value`. Это приведет к 500 ошибке для клиента или падению процесса (если ошибка не перехвачена корректно).
- **Решение:** Добавить проверку на пустоту массива перед вызовом `reduce`:
  ```typescript
  if (detailedProducts.length === 0) return null; // или пустой объект
  ```

### Проблема: Алгоритмическая сложность O(N^2) в `ProxyService`
В методе `checkAll` внутри цикла перебора результатов используется поиск индекса:
```typescript
const index = results.indexOf(result);
```
- **Риск:** При проверке большого списка прокси (например, 1000+) это создает лишнюю нагрузку на CPU, так как для каждого элемента выполняется повторный проход по массиву.
- **Решение:** Использовать обычный цикл `for` с индексом `i` для одновременного доступа к `proxiesData[i]` и `results[i]`.

## 3. Рекомендации по коду

- **Типизация:** В коде часто встречается `@ts-ignore` (например, в конструкторах сервисов) и приведение типов через `<string>`. Это снижает надежность кода. Рекомендуется определить точные интерфейсы для параметров конструктора.
- **Обработка ошибок:** В `BrowserService` используется ручная сериализация JSON для сохранения состояния в Redis. Стоит рассмотреть использование готовых библиотек или более надежных механизмов сериализации, чтобы избежать ошибок при изменении структуры данных.
