# Code Review: Marketplace Parser Service

Ниже представлен подробный code review проекта `marketplace-parser-service`. Обзор разбит на 4 основные категории: Архитектура, Качество кода (SOLID/KISS/DRY), Баги и Надежность, Производительность.

## 1. Архитектура и Дизайн

### Проблемы:
*   **Ручная регистрация парсеров (Open/Closed Principle Violation):**
    *   В `src/App.ts` происходит ручной импорт и регистрация классов парсеров (`WildBerriesParser`, `OzonParser` и т.д.).
    *   Это нарушает принцип открытости/закрытости: для добавления нового парсера нужно модифицировать `App.ts`.
    *   **Решение:** Используйте автоматическое обнаружение (auto-discovery) или конфигурацию, чтобы регистрировать парсеры динамически. Можно сканировать директорию `parsers/` или использовать возможности Awilix для загрузки модулей.

*   **Dependency Injection (DI) Inconsistency:**
    *   В `src/services/parser/ParserRegistry.ts` создается *вложенный* DI контейнер (`this.container = createContainer()`). Это усложняет управление зависимостями. Если парсеру понадобится сервис из основного контейнера (например, `Logger` или `Config`), его придется пробрасывать вручную.
    *   Использование `// @ts-ignore` в конструкторах скрывает проблемы с типами зависимостей.
    *   **Решение:** Регистрируйте парсеры в основном контейнере (в `src/utils/container.ts`) и инжектируйте их в `ParserRegistry` как список или карту. Избавьтесь от вложенного контейнера.

*   **Сильная связность `App.ts`:**
    *   Класс `App` знает слишком много о конкретных реализациях (`WildBerriesParser`, `OzonParser`). Он должен зависеть только от абстракций или `ParserRegistry`.

## 2. Качество кода (SOLID, KISS, DRY)

### Проблемы:
*   **Дублирование кода (DRY Violation):**
    *   В `OzonParser.ts` метод `findProduct` полностью дублирует реализацию из базового класса `MarketPlaceParser.ts`.
    *   В методах `fetchProducts` разных парсеров есть схожая логика задержек, скриншотов и обработки ошибок.
    *   **Решение:** Удалите переопределение `findProduct` в `OzonParser`. Вынесите общую логику (например, навигацию с ожиданием, снятие скриншотов для отладки) в методы базового класса или утилиты.

*   **Hardcoded Values (KISS Violation):**
    *   Таймауты (`5000`, `3000`), селекторы, URL-адреса и пути к скриншотам (`process.cwd()/screenshots/...`) захардкожены.
    *   Геолокации и User-Agent в `BrowserService` (метод `generateFingerprint`) захардкожены.
    *   **Решение:** Вынесите конфигурацию (URL, таймауты, флаги отладки) в файлы конфигов (`src/configs/config.ts`) или переменные окружения.

*   **Типизация (TypeScript):**
    *   Многочисленные `// @ts-ignore` и `any`.
    *   Приведение типов `Number.parseFloat(<string>...)` без проверки на `null/undefined` может привести к `NaN` или ошибкам в рантайме.

## 3. Баги и Надежность (Critical)

### Критические проблемы:
1.  **Утечка ресурсов в `BrowserProxyService` (Memory Leak):**
    *   В методе `checkProxy` (`src/services/proxy/BrowserProxyService.ts`) создается контекст браузера `testContext`.
    *   Если `page.goto` падает с ошибкой (что вероятно для плохих прокси), управление переходит в блок `catch`.
    *   В блоке `catch` **не вызывается** `testContext.close()`. Контекст остается висеть в памяти. При частых проверках это приведет к падению сервера (OOM).
    *   **Решение:** Перенесите `await testContext.close()` в блок `finally`.

2.  **Ненадежная очистка в `BrowserService`:**
    *   В методе `closeContext`, если `data.context.close()` выбросит исключение, строка `this.contexts.delete(userId)` не выполнится. Ссылка на объект останется в `Map` навсегда.
    *   **Решение:** Оберните закрытие контекста в `try/catch` или используйте `finally` для удаления из `Map`.

3.  **Swallowing Errors (Подавление ошибок):**
    *   В `src/index.ts`: `try { await app.init(); } catch (e: any) { }`. Ошибки при старте приложения игнорируются. Если база данных не подключится, приложение "зависнет" или упадет позже с непонятной ошибкой.
    *   В `MarketPlaceParser.safeFetchText`: Возвращает `null` при любой ошибке. Это удобно, но может скрыть проблемы с изменением верстки сайта.
    *   **Решение:** В `index.ts` обязательно логируйте ошибку и делайте `process.exit(1)`.

4.  **Скриншоты в Production:**
    *   Парсеры сохраняют скриншоты на диск (`page.screenshot`). В продакшене это быстро забьет дисковое пространство (inode limit или disk usage).
    *   **Решение:** Отключите скриншоты по умолчанию. Включайте их только при наличии флага `DEBUG_SCREENSHOTS=true`.

## 4. Производительность (Performance)

### Критические проблемы:
1.  **Unbounded Concurrency в `BrowserProxyService` (DoS самого себя):**
    *   В методе `checkProxies` используется `Promise.allSettled(proxiesData.map(...))`.
    *   Это запускает проверку **всех** прокси одновременно. Если у вас 100 прокси, сервис попытается открыть 100 экземпляров браузера (или контекстов) одновременно. Это гарантированно убьет CPU и RAM сервера.
    *   **Решение:** Используйте библиотеку `p-limit` или реализуйте chunking (разбиение на пачки), чтобы проверять не более 5-10 прокси одновременно.

2.  **Последовательный скрейпинг в `OzonParser`:**
    *   В методе `fetchProducts` используется цикл `for` с `await` внутри:
        ```typescript
        for (let i = 0; i < count; i++) {
            products.push(await this.parseProduct(card));
        }
        ```
    *   Это очень медленно. Если на странице 30 товаров и каждый парсится 0.1с, это добавит 3 секунды задержки.
    *   **Решение:** Используйте `Promise.all` для параллельного парсинга карточек, как это сделано в `WildBerriesParser`.

3.  **Resource Usage в `ProductAggregatorService`:**
    *   Создается новая страница (`context.newPage()`) для каждого парсера при каждом запросе поиска.
    *   Если запросов много, это создает большую нагрузку на браузер.
    *   **Решение:** Рассмотрите возможность переиспользования страниц или использования одного контекста с несколькими вкладками более экономно. Убедитесь, что `page.close()` вызывается всегда (сейчас это в `finally`, что хорошо).

---

## Итоговое резюме
Проект имеет неплохую структуру, но страдает от проблем с управлением ресурсами (утечки памяти, неограниченная конкурентность) и архитектурных недочетов в DI. **Критические баги в `BrowserProxyService` требуют немедленного исправления перед запуском в продакшн.**
