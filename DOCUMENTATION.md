# Описание архитектуры и взаимодействия микросервиса `marketplace-parser-service`

## Введение
Данный документ представляет собой подробное описание архитектуры, паттернов проектирования и механизмов взаимодействия микросервиса `marketplace-parser-service`. Целевой аудиторией документа являются ведущие разработчики (Team Leads) и архитекторы программного обеспечения.

## 1. Назначение микросервиса
Микросервис `marketplace-parser-service` предназначен для автоматизированного сбора (парсинга) данных о товарах с различных российских маркетплейсов (Ozon, Wildberries, Yandex Market, MegaMarket, Magnit Market, Alikson). Основная бизнес-задача заключается в предоставлении унифицированного интерфейса для поиска и получения детальной информации о товарах в условиях активного противодействия парсингу со стороны целевых площадок.

## 2. Архитектура проекта
Проект построен на базе микросервисной архитектуры с использованием платформы Node.js (TypeScript) и фреймворка Express для обработки HTTP-запросов (REST API).

### 2.1. Используемый стек технологий
*   **Язык программирования:** TypeScript
*   **Среда выполнения:** Node.js v22.22.0
*   **Веб-фреймворк:** Express v5.2.1
*   **Браузерная автоматизация:** Playwright, `playwright-extra`, `puppeteer-extra-plugin-stealth`, `camoufox-js` (для обхода систем защиты от ботов).
*   **Брокер сообщений:** RabbitMQ (`amqplib`)
*   **Кэширование и хранение состояний:** Redis (`ioredis`)
*   **База данных (ОРМ):** PostgreSQL + Prisma ORM (`@prisma/client`, `@prisma/adapter-pg`)
*   **Внедрение зависимостей (DI):** Awilix

### 2.2. Паттерны проектирования
В проекте строго соблюдается разделение ответственности (Separation of Concerns) с использованием следующих архитектурных паттернов:

1.  **Service-Repository:** Слой доступа к данным (репозитории) абстрагирован от бизнес-логики (сервисов). Сервисы расположены в директории `src/services`, а логика работы с базой данных — в `src/repo` (или делегируется Prisma).
2.  **Dependency Injection (Внедрение зависимостей):** Управление жизненным циклом компонентов и их зависимостями осуществляется с помощью IoC-контейнера `Awilix`. Используется режим `InjectionMode.PROXY`, что позволяет передавать все зависимости в конструктор классов в виде единого деструктурированного объекта (напр., `constructor({ webServer, redisClient, ... })`). Это облегчает тестирование и снижает связность компонентов.
3.  **Factory (Фабрика):** Применяется для создания логгеров (`loggerFactory`) и регистрации парсеров (`ParserRegistry`).
4.  **Strategy (Стратегия):** Различные маркетплейсы имеют уникальную логику парсинга. Реализован базовый абстрактный класс `MarketPlaceParser`, от которого наследуются конкретные реализации (напр., `WildBerriesParser`, `OzonParser`).

### 2.3. Структура проекта
*   **`src/controllers`**: Контроллеры для обработки HTTP-запросов REST API (BrowserController, ProductController, ProxyController).
*   **`src/routes`**: Определение маршрутов Express и привязка их к методам контроллеров (`setupRoutes()`).
*   **`src/services`**: Ядро бизнес-логики микросервиса (парсинг, управление прокси, управление браузерами).
    *   **`src/services/parser/parsers`**: Имплементации парсеров для каждого маркетплейса.
*   **`src/infrastructure`**: Инфраструктурные компоненты (напр., подключение к RabbitMQ).
*   **`src/jobs`**: Фоновые задачи (cron jobs), такие как планировщик прокси (`ProxyScheduler`).
*   **`src/handlers`**: Обработчики специфических событий.

## 3. Взаимодействие микросервисов (Message Broker Topology)
Взаимодействие с внешними системами (в частности, с **API Gateway**) построено на базе асинхронного обмена сообщениями через **RabbitMQ**.

### 3.1. Топология RabbitMQ
В проекте реализована надежная система очередей с поддержкой механизма Dead Letter Queue (DLQ) для обработки сбоев:
*   **Exchanges:**
    *   `tasks` (direct) — входящие задачи от API Gateway.
    *   `marketplace.parser` (topic) — исходящие результаты парсинга.
    *   `tasks.dlx` (direct) — Dead Letter Exchange для упавших задач.
*   **Queues (Задачи):**
    *   `tasks.preview` — очередь задач на получение превью товаров (быстрый поиск).
    *   `tasks.detailed` — очередь задач на получение детальной информации о товаре.
    *   `tasks.preview.dlq` / `tasks.detailed.dlq` — очереди для задач, завершившихся с ошибкой.
*   **Queues (Результаты):**
    *   `marketplace.parser.preview`
    *   `marketplace.parser.detailed`

### 3.2. Поток выполнения (Workflow)
1.  **API Gateway** публикует задачу в обменник `tasks` с ключом маршрутизации `tasks.preview` или `tasks.detailed`. В задаче передается уникальный `sessionId`.
2.  `TaskConsumer` сервиса-парсера извлекает задачу из очереди (с механизмом `prefetch` для контроля нагрузки).
3.  `TaskHandler` делегирует выполнение `ProductAggregatorService`.
4.  После успешного парсинга `ParserPublisherService` публикует результат в обменник `marketplace.parser`, используя `sessionId` в качестве ключа маршрутизации.
5.  **API Gateway**, предварительно подписавшийся на временную очередь, привязанную к этому `sessionId` (`bindResultQueues`), получает готовые данные. В случае ошибки отправляется уведомление об ошибке парсинга.

## 4. Масштабируемость, отказоустойчивость и безопасность

### 4.1. Горизонтальное масштабирование
Архитектура приложения разработана с учетом возможности запуска множества реплик микросервиса (Stateless Workers). Вся логика управления состоянием вынесена во внешние хранилища:
*   **Redis:** Хранит сессии пользователей (Storage State Playwright), отпечатки браузеров (Fingerprints) и выполняет роль кэша продуктов (`ProductCacheService` с настройками TTL).
*   **PostgreSQL:** Хранит конфигурации и статусы прокси-серверов.
RabbitMQ обеспечивает равномерное распределение задач между запущенными экземплярами сервиса (Round-Robin).

### 4.2. Конкурентность и защита от состояния гонки (Race Conditions)
Микросервис спроектирован для обработки большого количества параллельных запросов.
*   **Изоляция контекстов:** `BrowserService` создает независимые контексты Playwright (`BrowserContext`) для каждой сессии (`sessionId`). Каждому контексту назначается уникальный отпечаток и, при необходимости, отдельный прокси-сервер (`ProxyService`).
*   **Асинхронные блокировки:** При операциях с разделяемыми ресурсами (например, выдача прокси-сервера из пула в PostgreSQL) используется транзакционная логика Prisma ORM, исключающая выдачу одного прокси двум разным сессиям одновременно.
*   **Ограничение параллелизма (Rate Limiting):** Для предотвращения исчерпания ресурсов (RAM/CPU) при запуске множества вкладок браузера применяется ограничение параллелизма (например, библиотека `p-limit`, указанная в зависимостях), а `channel.prefetch(10)` в RabbitMQ ограничивает количество одновременно обрабатываемых задач одним воркером.

### 4.3. Обход анти-фрод систем (Anti-Bot Protection)
Многие маркетплейсы (особенно Yandex Market и MegaMarket) используют строгие системы защиты. Для их обхода сервис реализует продвинутые техники:
1.  **Генерация отпечатков (Fingerprints):** `FirefoxUserAgentGenerator` и `BrowserService` динамически генерируют уникальные параметры для каждого контекста: User-Agent (с учетом ОС), геолокацию, разрешение экрана (Viewport), язык (ru-RU) и часовой пояс (Europe/Moscow).
2.  **Эмуляция реального пользователя:** Использование `camoufox-js` и плагинов `playwright-extra` со stealth-режимом для подмены параметров WebGL, Canvas и скрытия следов автоматизации (свойства `navigator.webdriver`).
3.  **Ротация и управление Proxy:** В `ProxyService` реализована система управления прокси. Каждому контексту назначается прокси из пула (с учетом статуса `ACTIVE`, `FAILED`, `SUSPENDED`). Сервис ведет учет неудачных попыток (`failedAttempts`) и использует `ProxyScheduler` для периодической проверки и ротации прокси.
4.  **Управление сессиями:** Сессии браузера сохраняются в Redis (`browser_context:{id}`) с заданным TTL. Это позволяет переиспользовать cookies и storage state для повторных запросов, снижая вероятность блокировки за частую авторизацию или подозрительную активность. Фоновые задачи (`node-cron`) очищают устаревшие контексты, предотвращая утечки памяти (Memory Leaks).

## Заключение
Микросервис `marketplace-parser-service` представляет собой высоконагруженную, масштабируемую систему, готовую к эксплуатации в production-среде. Использование паттернов проектирования, надежного брокера сообщений RabbitMQ и продвинутых механизмов обхода блокировок Playwright позволяет обеспечивать стабильный сбор данных с маркетплейсов в рамках распределенной архитектуры предприятия.
